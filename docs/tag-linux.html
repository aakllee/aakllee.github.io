PiCMCA - Part #1

The Pi Compute Module Cardbus Adapter: PiCMCA, is a Cardbus (PCMCIA) module that uses the USB lanes of the Cardbus interface as a low-latency network connection between the host system and a Raspberry Pi compute module system-on-a-chip.

This allows for the host laptop to act as a thin client for the Raspberry Pi, thus allowing vintage computers to act as modern computing devices.

Why?

I am the happy owner of a Thinkpad x61, which is now a grand 14 years old. It is, of course, beginning to show its age, with its fans loud as a microwave, and a battery with the lifespan of an elderly mayfly.

For all that, it is still in excellent condition, so I wanted to give it another chance at life. And I would assume that there are many others similarly attached to their old computers who could stand to benefit - not to mention the potential reduction in e-waste.

I then came across the Thinkpad x62/x63: an extreme modification to the x61 which supplants the original motherboard with an entirely custom-built motherboard with modern components.

Sadly, these were very expensive, and available for purchase only by wiring money to various individuals overseas, who would then mail the motherboard over to you (from anecdotal data, usually taking several months). And difficulties with customs and legalese etc. make things really quite dubious.

So, I did the natural thing, and decided to take matters into my own hands.

Plan

In the first place, I didn't need the full processing power of a mobile 5GHz Intel CPU with integrated graphics and a 4K LCD screen - far from it; I needed only something I could use to browse the web, comfortably write up some documents, watch some films. And not also perform a secondary function as a portable stove. So if I could implant the Pi compute module 4 into the existing internals, that would be a quite apposite solution.

The Pi would use the Thinkpad SSD for storage using a technology such as NFS or SSHFS, since the SD card/eMMC is unsuitable for anything more than the OS. Ideally I'd have the entire root filesystem on the Thinkpad using PXE/netbooting, but alas my proficiency in wizardry extends only so far. It might be possible to have the boot partition on the SD card then mount root as NFS, however this is an optimisation problem so to be tackled later on.

It is relatively messy to have the files on the client, but unlike in an OS like Plan9, we can't just set each as CPU server and File server. I was considering setting up Plan9, but felt that it would have hindered usability a little too much.

The plan was to design a standard PCB for the Pi CM4 (hence the name of the project), however I soon came to realise, reading the specification, that the height of a PCMCIA card is a quite strict 5mm. A Pi CM4 on the other hand, is 4.78mm tall.

So looking into what I could do there, I came across flex PCBs and extra thin FR4 PCBs, which have a potential depth of less than 0.20mm.

This seemed perfect, however, I could not find any detailed specification for the depth of the CM4 connector itself that would be soldered to the PCB, which if too tall would have been very sad indeed. I did find that the "mounted stacking height" of the connector would be 1.5mm, however I was unsure if this included the connector on the CM4 itself - in which case the total height would be 5.11mm…

So I assumed that tolerances wouldn't be that tight and went ahead anyway.

Proof of Concept

To begin with I used an off-the-shelf carrier board which breaks out the USB lanes, so that I would be able to test that my software stack would work in practice, before committing to designing my own using the Cardbus interface.

This took some quite extensive configuration, but I managed to make things work, running Alpine Linux on both the Pi and the laptop. Alpine is inherently a very small and efficient operating system, owing to have been designed for usage in embedded systems.

I initially considered using X-forwarding, however this would have resulted in needing to run two separate X11 servers, which would have been pointlessly inefficient.

But eventually what I found was that the solution was in the name - X11 servers being servers, surely meant that I could run the X11 server on the Pi and then use the laptop as solely a client. And this led me to XDMCP - a protocol for opening remote X sessions.

So I installed X11 on the Pi, for which Alpine provides a very convenient setup-xorg-script, and set up XDMCP, thus giving me a fully functional proof of concept.

Hardware design

Plan

I was able to use the regular Pi 4B in the proof of concept via USB gadget mode, which allows one to run the Pi headlessly, using the USB interface as a network connection to transfer data. In theory, I saw two paths to mimic this same connection over the Cardbus interface:

  1. Bridge Cardbus (PCI) to USB

    I looked at existing Cardbus-USB adapters and found that they all seemed to use the same chip: the VT6212/VT6212L. This seemed promising, although I was running into space constraints with fitting the 20x14mm footprint alongside the CM4.

  2. Bridge Cardbus (PCI) to Ethernet

    The cards I found for these seemed distinctly more complex, and I found that the gains in bandwidth would be largely redundant for the throughput needed by XDMCP (~100mbit), for which even USB1.0/Low-Speed should be sufficient (~1.5Mbps). So USB was looking much more pleasant to work with.

  3. Connect the compute module directly to the Cardbus interface.

    This seemed most efficient, and would be potentially simpler in terms of circuit design, but would likely have involved writing device drivers.

The USB route seemed most simple, so I am moving forward with that plan, but should there be some fatal flaws with e.g size constraints, I may be forced to consider other options.

I've neither designed a PCB nor used ICs before, so I was a little lost to begin with. I decided to acquire a Cardbus-USB adapter card that I could open up to reverse engineer.

AMD Vega 10 Fan Control on GNU Guix System

When I first built my PC, I installed GNU Guix System as my primary OS - a distribution of Linux with fully functional package management and system configuration. Guix System is similar to NixOS in this sense, but uses Scheme (a Lisp) instead of Nix. This allows one to write configurations without needing to learn the arcane and obscure peculiarities of Nix, so was extremely appealing to me.

And why not a normal distribution of Linux? Well… I didn't see any massive advantages to a normal distribution over Guix, so Guix it was.

And it's been over a year and it has served me well - I haven't needed to reinstall the system at all, practically everything just works.

Hardware

My PC is built in an ITX case (the Louqe Ghost S1) so there were some space constraints I had to work around when choosing parts:

  • ASRock B550m-itx/ac
  • AMD Ryzen 5600G
  • AMD Radeon Pro v320
  • Corsair SF750
  • Corsair Vengeance LPX 2x8GB
  • Noctua NH-L12 Ghost S1 Edition

But with everything together, it runs almost silently, staying cool and out of mind.

Except the GPU.

Now you may be wondering: Aaron, what in God's name is that GPU supposed to be? v320? That's not something you can even buy.

And no, the card was never made commercially available to consumers, they were bespoke cards made specifically for Google Stadia development machines, though in reality they are essentially typical Vega 10 cards with a shiny coat of YInMn blue. I bought one in a Japanese auction while the Yen was low, and it turned out quite cheap indeed.

Point is, the card ran weirdly hot, drawing a lot of power. So it was the noisiest part of the whole PC. So I wanted to fix that.

Solution

There were existing scripts for fan control on Linux (eg amdgpu-fan), some written in Python, some in C, some in Bash. So I adapted these into Scheme, so as to integrate into Guix System, and added a function to also undervolt the card: 

#!/usr/bin/env -S guile
!#

(use-modules (ice-9 textual-ports))

;; path to gpu device folder
(define d "/sys/class/drm/card0/device/")
(define h (string-append d "hwmon/hwmon2/"))

;; amdgpu-fancontrol constants
(define y 6000) ; in mK
(define s 1) ; in s
(define p (string-append h "pwm1"))
(define f (string-append h "pwm1_enable"))
(define t (string-append h "temp1_input"))

;; undervolt/overclock constants
(define c (string-append d "pp_od_clk_voltage"))
(define e (string-append d "power_dpm_force_performance_level"))

(define l 0)

(define (w f s)
  "writes str to file"
  (let ((o (open-file f "w")))
    (display s o)
    (newline o)
    (close o)))
(define (r f)
  "returns file contents as string"
  (get-string-all (open-file f "r")))

;; method used to set clock speeds and voltages
(define (set-clks a sm)
  "lst should be in form L C V L C V L C V..."

  (w e "manual")

  ;; for each L C V
  (for-each
   (lambda (l)
     ;; print to console
     (display "writing to clk: ")
     (display (string-join (cons sm l) " "))
     (newline)
     ;; write to file
     (w c (string-join (cons sm l) " ")))

   (parse-clks a))

  (w c "c")
  (display (string-concatenate (list "Finished setting " sm "clk")))
  (newline))


;; methods used for parsing input
(define (list->alist l)
  "Converts list (a b c d ...) to ((a b) (c d) ...)"
  (if (< (length l) 2)
      '()
      (cons (cons (string->number (string-trim-both (car l)))
                 (string->number (string-trim-both (cadr l))))
           (list->alist (cddr l)))))
(define (list-split l p)
  "returns (before . after) where before is the sublist before 
pred, and after is the sublist after pred. Assumes pred is unique."
  (define (until l p)
    (if (or (null? l)
           (string=? (car l) p))
        '()
        (cons (car l) (until (cdr l) p))))
  (cons (until l p) (reverse (until (reverse l) p))))
;; probably should merge this into list->alist with a numerical argument
(define (parse-clks a)
  (if (< (length a) 3)
      '()
      (cons (list (car a) (cadr a) (caddr a))
           (parse-clks (cdddr a)))))

;; checks for read/write access
(define check-permissions
  (and (access? p W_OK)
       (access? f W_OK)
       (access? t R_OK)
       (access? c W_OK)
       (access? e W_OK)))


;; sets fan mode
(define (set-fanmode m)
  (display (string-append "setting fan mode to " m))
  (w f mode))
;; sets pwm
(define (set-pwm m)
  "pwm is the value to set to"
  (let ((n m)
        (o (string->number (string-trim-both (r p)))))
    (display (string-concatenate
             (list "current pwm: " (number->string o)
                   ", requested to set pwm to " (number->string n))))
    (newline)
    (if (not (string=? (string-trim-both (r f)) "1"))
        (let ()
         (display "fanmode not set to manual. Setting now.")
         (newline)
         (set-fanmode "1")))

    ;; set temp iff:
    ;;  temperature at last pwm change is unset (0)
    ;;  or current temperature > temperature at last pwm change
    ;;  or t + y <= temperature at last pwm change
    (let ((t (string->number (string-trim-both (r t)))))
      (if (or (= l 0)
             (> t l)
             (> l (+ t y)))
         (let ()
           (display "current temp: " )
           (display t)
           (newline)
           (display (string-append "temp at last change was " (number->string l)))
           (newline)
           (display (string-append "changing pwm to " (number->string m)))
           (newline)
           (w p n)
           (display "last: ")
           (display l)
           (newline)
           (display "setting last to: ")
           (display (r t))
           (set! l
                 (string->number
                  (string-trim-both
                   (r t)))))))))
(define (interpolate-pwm m)
  "m is the alist of (temperature . pwm) pairs"
  (define (get-next-interpolation-pairs t m)
    "returns a pair of the next two temps and pwms to use"
    (if (> (caadr m) t)
        (cons (car m) (cadr m))
        (get-next-interpolation-pairs temp (cdr m))))
  ;; let t be the integer value of the contents of the temperature file
  (let ((t (string->number (string-trim-both (r t)))))
    ;; set pwm to the clamped linear interpolation between the values of m
    (set-pwm 
     (cond ((> (caar m) t) ; if temp <= min temp, set to min temp
           (caar m))
          ((< (caar (last-pair m)) t) ; if temp >= max temp, set to max temp
           (caar (last-pair m)))
          (else (let* ((v (get-next-interpolation-pairs t m)))
                  (+ lower-pwm
                     ;; / would return a rational number so use quotient
                     (quotient (* (- t (caar v))
                                  (- (cddr v) (cdar v)))
                                (- (cadr v) (caar v))))))))))
;; method used to start fan daemon
(define (run-daemon a)
  "runs the daemon"
  (while #t
        (interpolate-pwm a)
        (sleep s)))
;; main method
(define (amduvocd a)
  "a are in the form and order:
TEMP PWM  TEMP PWM...  -s LVL SCLK V  LVL SCLK V...  -m LVL MCLK V  LVL MCLK V... 
because parsing inputs properly is a waste of time. If your a aren't formatted 
correctly, something will probably go wrong, so, don't do that"
  (if (not (access? h F_OK))
      (display (string-concatenate
                (list "error: " h
                     " does not exist. You may need to edit this file to change the path. "
                     "(I may implement customisation at some point, but probably never)"))))
  (if check-permissions
      ;; if sufficient permissions, set values
      (let* ((a (list-split a "-s"))
            (fan-a (list->alist (cdar a)))
            (clk-a (list-split (cdr a) "-m"))
            (sclk-a (car clk-a))
            (mclk-a (cdr clk-a)))
        (display "Fan pwms:")
        (newline)
        (display fan-a)
        (newline)
        (display "SCLK:")
        (newline)
        (display sclk-a)
        (newline)
        (display "MCLK:")
        (newline)
        (display mclk-a)
        (newline)
        (newline)
        (set-clks sclk-a "s")
        (set-clks mclk-a "m")
        (run-daemon fan-a)
        (status:exit-val 0))

      ;; return error code 1 if invalid permissions
      (let ()
        (display "Error: insufficient permissions. Are you root?")
        (status:exit-val 1))))
;; execute main method
(amduvocd (command-line))

Results

It works. Card runs quiet, draws less power. Easy to customise if I ever need more performance for some reason. No particular issues I have noticed yet, although always on the lookout.
